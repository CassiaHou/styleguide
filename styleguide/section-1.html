1Style Rules

Here are detailed all the style rules for the web application.

1.1Depth

Getting a deeper look into the styles of shadows and positioning. The first thing we need to understand is the concept of stacked material (see the Google guidelines ).

Keep the number of stacked layers to a minimum and the total number of layer below 3 (including the background). If a raised element lives inside of another raised element it should have a higher material-depth number (i.e a panel with a depth of 1 can have one panel with a depth of 2 above (inside) it.)

Elements floating above the main layout (such as context menus) should have the highest material-depth number possible (2).

Note: the 'spacer' class is for styleguide demo purpose only.

Example
Background
Layer 1
Layer 2

Stacked 

<div class="spacer"> Background </div>
<div class="spacer material-depth-1"> Layer 1 </div>
<div class="spacer material-depth-2"> Layer 2 </div>
<div class="spacer material-depth-1">
<p class="spacer material-depth-1"> Stacked </p>
</div>

1.2.1Trench

We only use it for user interaction on small details.

The trench works much like the depth, however since it gives the illusion of a layer behind the current piece of material it should not be used on larger areas, as this will create raised "frames" around the trench.

Note: the 'spacer' class is for styleguide demo purpose only.

1.6Buttons

When using buttons, avoid non specific actions as a label. For example, "OK", "GO", "ACCEPT" are not specific while "SAVE USER" and "GO ON DASHBOARD" are. This helps the user understand what the action will do and what object it will effect. For more details take a look at the material design buttons

The primary version of the button should be used for the primary action (i.e the one the user is most likely to take) in that view. By that logic there should normally only be one per screen.

There are different types of buttons, however try to use one type of button per container. Only mix button types when you have a good reason to do so.

1.6.1Flat buttons

The flat buttons are 'printed' on the material and only show its area when hovered, it uses capital letters to differentiate it from a link.

Example

<button class="wfm-btn wfm-btn-invis-disabled">Disabled</button>

1.8Tables

Tables are used to give a overview of data, however, following the Main concepts of the styleguide, only relevant data should be displayed. In the cases where that is not possible, please use infinite scrolling or pagination to keep the page lightweight. If a table does take time to be loaded, a loading bar should be displayed during this time to indicate to the user that it will take some time and that the table is not broken.

A table should be displayed in alphabetical order by default, unless there is an obvious way the user would want the data sorted. Even so, the ability to sort and search within the table is most welcome. If the table can be sorted, it should have filled arrows in the header indicating the sorting direction (down for descending and up for ascending), clicking the header should change the sorting direction and the arrow direction.

Cells containing numbers are right aligned using the 'number-cell' class to differentiate them from text content.
If you need to display bigger forms of data or need more complex actions such as importing/exporting, multi select or the ability to hide columns - take a look at the 'Big table' in chapeter 6 - Scalability.

1.10Notices

Notices indicate to the user that an event or action has taken place or that something is stopping the current action. There are several different markers to make the type of notice easy to spot:

  • Success: Shown to indicate that an action has been done successfully.
  • Info: Should be used to give the user hints and information.
  • Details: These are never shown alone, but rather beneath one of the other markers.
  • Warning: Should be used to alert the user that something have to be done before they can continue.
  • Error: Should only be triggered when something goes wrong in the background.

Note: the 'notice-spacer' class is for styleguide demo purpose only.

A notice should either self-destruct or be dismissable to free up space.

Example
Success: User is saved successfully.
Info: A user logged out.
I was born to show details.
Warning: Press refresh as the data was changed by another user.
Error: Something exploded so fix it.
<div class="alert-success notice-spacer"><i class='mdi mdi-thumb-up'></i> Success: User is saved successfully.</div>
<div class="notice-group">
<div class="alert-info notice-spacer"><i class='mdi mdi-information'></i> Info: A user logged out.</div>
<div class="alert-detail notice-spacer">I was born to show details.</div>
</div>
<div class="alert-warning notice-spacer"><i class='mdi mdi-alert'></i> Warning: Press refresh as the data was changed by another user.</div>
<div class="alert-error notice-spacer"><i class='mdi mdi-alert-octagon'></i> Error: Something exploded so fix it.</div>

1.16Tabs

Tabs are used to change big content areas and are a great way to divide a single task into smaller pieces. They are typicaly located at the top of the page or a big panel instead of a header. One important thing to remember is to order the tabs in a way that makes sense to the workflow. Also, avoid having nested tabs or several tab -groups close together to avoid confusion.

Example
